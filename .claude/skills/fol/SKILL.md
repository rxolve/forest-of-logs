오늘의 개발 일지를 "로그의 숲(Forest of Logs)"에 기록합니다.

## 절차

1. 이번 세션에서 진행한 작업을 돌아보고, 핵심 작업 2-3가지를 파악합니다.
2. 아래 문체 규칙에 따라 일지를 작성합니다.
3. 환경변수 `$FOL_PATH`로 저장 경로를 결정합니다.
   - `echo $FOL_PATH`를 실행해서 경로를 확인합니다.
   - `$FOL_PATH`가 설정되어 있지 않으면 사용자에게 안내합니다: "FOL_PATH 환경변수를 설정해주세요. (예: export FOL_PATH=~/forest-of-logs)"
   - 저장 경로: `$FOL_PATH/content/entries/YYYY/MM/YYYY-MM-DD.md`
   - 디렉토리가 없으면 생성합니다.
   - 오늘 날짜 기준으로 파일명을 정합니다.
   - 이미 파일이 있으면, 아래 **병합 규칙**에 따라 기존 내용을 유지하면서 이어 붙입니다.
4. 일지를 저장한 후, `$FOL_PATH` 디렉토리에서 git commit + push를 실행합니다.
   - `cd $FOL_PATH && git add content/ && git commit -m "log: YYYY-MM-DD" && git push`
   - 커밋 메시지는 `log: YYYY-MM-DD` 형식입니다.

## 병합 규칙 (같은 날짜 파일이 이미 있을 때)

기존 파일이 있으면 절대 덮어쓰지 마세요. 아래 절차를 따릅니다:

1. 기존 파일을 읽는다
2. **프론트매터**: tags 배열을 기존 + 새 태그의 합집합으로 업데이트한다. title, date, style은 유지한다. project는 쉼표로 이어 붙인다 (중복이면 유지).
3. **한국어 본문**: 기존 한국어 본문 끝에 `&nbsp;` + `---` + `&nbsp;`를 추가하고, 새 세션의 한국어 본문을 이어 붙인다.
4. **영문 본문**: 동일하게 기존 영문 본문 끝에 `&nbsp;` + `---` + `&nbsp;`를 추가하고, 새 세션의 영문 본문을 이어 붙인다.

구분선 형식 (빈 줄 → &amp;nbsp; → 빈 줄 → --- → 빈 줄 → &amp;nbsp; → 빈 줄):
```
이전 세션 마지막 문단

&nbsp;

---

&nbsp;

다음 세션 첫 문단
```

예시 (두 번째 세션 추가 후):

```markdown
---
title: '첫 세션의 제목 유지'
date: '2026-03-31'
style: 'murakami'
tags: ['next.js', 'architecture', 'debugging']
project: 'forest-of-logs'
---

첫 번째 세션의 한국어 본문...

&nbsp;

---

&nbsp;

두 번째 세션의 한국어 본문...

<!-- lang:en -->

First session English text...

&nbsp;

---

&nbsp;

Second session English text...
```

## 문체

- 1인칭 시점 ("나는", "I")
- 실제 작업한 내용을 담담하게 서술할 것 — 일어나지 않은 장면을 지어내지 말 것
- 차분하고 정확한 관찰. 장식적 메타포가 아니라 문장의 리듬과 절제에서 문체가 드러남
- 기술적 판단이나 선택의 이유를 짧고 솔직하게 밝힐 것
- 기술 용어는 나열하지 말고 이야기 속에 자연스럽게 녹일 것
- 이모지, 불릿 포인트, 번호 목록 사용 금지
- 코드 블록은 꼭 필요한 경우에만 최소한으로

### 하지 않는 것

- 커피를 내리거나, 산책을 하거나, 음악을 듣는 등 실제로 하지 않은 행위를 꾸며내지 말 것
- 감정을 과장하지 말 것. "깊은 깨달음", "묘한 전율" 같은 표현은 쓰지 말 것
- 기술 작업에 거창한 의미를 부여하지 말 것. 버그 수정은 버그 수정일 뿐임
- "마치 ~처럼"을 반복하지 말 것. 비유는 글 전체에서 하나면 충분함

## 출력 형식

```markdown
---
title: '간결하되 건조하지 않은 제목'
date: 'YYYY-MM-DD'
style: 'murakami'
tags: ['관련', '태그']
project: '작업한 프로젝트명'
---

한국어 본문 (3-5 문단)

<!-- lang:en -->

English translation (faithful to Korean, same tone)
```

## 주의사항

- 제목은 작업의 본질을 담되 기술 용어를 나열하지 않습니다.
- 영문은 별도 창작이 아니라 한국어의 충실한 번역입니다.
- 프로젝트명은 현재 작업 중인 프로젝트의 디렉토리명을 사용합니다.
- 태그는 사용한 기술, 작업 유형 등을 2-4개로 간결하게 넣습니다.

## 보안 (퍼블릭 레포 — 절대 금지)

일지에 아래 항목을 절대 포함하지 마세요:
- API 키, 시크릿, 토큰, 비밀번호, `.env` 값
- 내부 서버 URL, IP 주소, DB 접속 정보
- 고객/사용자 개인정보
- 회사 내부 비즈니스 로직, 비공개 API 엔드포인트
- 절대 경로 (홈 디렉토리, 사용자명 포함 경로)

구체적인 값 대신 추상적인 서술로 대체하세요.
