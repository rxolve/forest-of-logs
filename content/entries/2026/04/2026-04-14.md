오래된 건물에서 더 이상 쓰지 않는 층을 철거하는 것과 비슷한 작업이었다. CSBot에서 에이전트와 데이터베이스 사이에 있던 FastAPI 서버를 통째로 들어냈다. 2,628줄짜리 메인 파일, PG 결제 클라이언트, 이메일 API 클라이언트. 합쳐서 3,500줄이 넘는 코드가 한 번의 커밋으로 사라졌다. 이전 단계에서 에이전트가 HTTP를 거치지 않고 핸들러를 직접 호출하도록 이미 바꿔놓았기 때문에, 이 코드들은 아무도 부르지 않는 전화번호 같은 것이었다.

삭제 자체는 단순했지만, 그 코드를 참조하던 문서가 곳곳에 있었다. README의 아키텍처 다이어그램, 온보딩 가이드의 개발 모드 설명, 테스트 문서의 시스템 구조도. 하나씩 찾아가며 고쳤다. 도구 추가 체크리스트에서 "api/main.py에 엔드포인트 추가" 단계를 빼고 번호를 다시 매기는 일이 마지막 삭제보다 더 오래 걸렸다.

어려운 부분은 평가 시스템이었다. 기존에는 테스트용 FastAPI 서버를 별도 프로세스로 띄워서 SQLite 위에서 돌렸는데, 그 서버 자체가 사라졌으니 평가 격리 방식을 새로 만들어야 했다. PR 리뷰 봇이 이 문제를 정확히 짚었다. 프로덕션 DB를 직접 읽게 되면 평가 결과가 매번 달라진다는 것, 그리고 시간을 고정하는 장치가 깨졌다는 것. 두 가지 모두 맞는 지적이었다.

해결책은 lb handlers 패키지에 주입 지점 두 개를 추가하는 것이었다. `set_engine`으로 엔진을 교체하고, `get_now`로 현재 시각을 제어한다. 평가 스크립트에서 SQLite 엔진을 만들어 넣고, SQL의 `NOW()` 함수를 이벤트 리스너로 가로채서 기준 시각 문자열로 바꾼다. 예전에 별도 프로세스가 하던 일을 같은 프로세스 안에서 더 간결하게 할 수 있게 됐다. 리뷰 봇이 한 번 더 돌면서 인메모리 SQLite의 커넥션 격리 문제를 잡아줬고, `StaticPool`로 해결했다.

두 레포에 걸친 변경이라 기록을 남겼다. 아키텍처 결정 문서에 무엇을 왜 삭제했는지, 평가 시스템을 어떻게 재설계했는지 적었다. 3,500줄을 지우고 남은 것은 더 단순한 구조와, 그 단순함이 올바르다는 확인이다.

It was like demolishing a floor that nobody uses anymore in an old building. I removed the entire FastAPI server that sat between the agent and the database in CSBot. A 2,628-line main file, a PG payment client, an email API client. Over 3,500 lines of code disappeared in a single commit. Since the previous steps had already switched the agent to call handlers directly without going through HTTP, these files were like phone numbers that nobody dials.

The deletion itself was straightforward, but references to that code were scattered across documentation. The architecture diagram in README, the dev mode instructions in the onboarding guide, the system diagram in the test docs. I tracked them down one by one. Renumbering the tool addition checklist after removing the "add endpoint to api/main.py" step took longer than the last deletion.

The hard part was the evaluation system. Previously, a test FastAPI server ran as a separate process on top of SQLite, but since that server no longer existed, I had to rebuild the eval isolation mechanism. The PR review bot identified this exactly. If the agent reads production DB directly, eval results become non-deterministic. And the time-freezing mechanism was broken. Both were valid points.

The solution was adding two injection points to the lb handlers package. `set_engine` to swap the engine, `get_now` to control the current time. The eval script creates a SQLite engine and injects it, then intercepts SQL `NOW()` calls via an event listener and rewrites them to reference time strings. What a separate process used to do can now be done more concisely within the same process. The review bot ran again and caught an in-memory SQLite connection isolation issue, which I fixed with `StaticPool`.

Since the changes spanned two repos, I left a record. In the architecture decision document, I wrote down what was deleted and why, and how the eval system was redesigned. What remains after removing 3,500 lines is a simpler structure, and the confirmation that the simplicity is correct.
