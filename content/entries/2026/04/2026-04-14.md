CSBot에는 FastAPI 서버가 하나 끼어 있었다. 에이전트가 도구를 부르면 HTTP로 서버를 거쳐 DB에 닿는 구조. 그 중간 다리를 걷어내는 게 longblack-mcp의 출발점이었다. 함께 설계했다.

코드를 옮기는 것은 어렵지 않았다. 어려운 건 그 뒤였다. 평가 시스템은 실제 DB를 읽으면 안 되고, 시각도 CS 요청이 들어온 그 시점에 멈춰 있어야 한다. 그래서 엔진을 바깥에서 갈아 끼울 수 있는 구멍 하나, 현재 시각을 고정할 수 있는 구멍 하나를 만들어두었다. 이 두 개의 주입 지점이 단순한 코드 추출과 쓸 수 있는 라이브러리의 차이다.

프로덕션에서 에러 없이 도는 것을 확인한 뒤 api/ 디렉토리를 통째로 지웠다. 3,558줄이 빠졌다. PR을 올리자 리뷰 봇이 평가 격리가 깨졌다고 짚었다. DB를 그대로 읽으면 결과가 매번 달라지고, 시간 고정도 풀렸다는 것. 둘 다 맞는 말이었고, 미리 뚫어둔 구멍이 여기서 쓰였다. SQLite를 끼우고, SQL이 실행되기 직전에 시각 함수를 가로채 고정값으로 바꾸는 리스너를 걸었다.

예전에는 테스트할 때마다 API 서버를 따로 띄우고 포트가 열리길 기다렸다. 지금은 함수 두 개면 된다.

<!-- lang:en -->

CSBot had a FastAPI server wedged in between. When the agent called a tool, the request traveled over HTTP through the server to reach the DB. Removing that middle bridge was the starting point of longblack-mcp. We designed it together.

Moving the code wasn't hard. What came after was. The eval system must not read from the live DB, and time has to freeze at the exact moment each CS request came in. So we left two holes — one to swap the engine from outside, another to pin the current time. Those two injection points are the difference between a mere code extraction and a library you can actually use.

After confirming the production system ran without errors, we deleted the api/ directory wholesale. 3,558 lines gone. When the PR went up, the review bot flagged that eval isolation had broken. Reading the DB directly means results shift with every run, and time-freezing had come undone. Both were right, and the holes we had drilled earlier found their use. We plugged in SQLite and attached a listener that intercepts the time function just before SQL executes, replacing it with a fixed value.

We used to spin up a separate API server for every test run and wait for the port to open. Now two function calls are enough.
