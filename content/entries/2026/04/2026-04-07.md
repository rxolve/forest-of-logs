artscii를 MCP 전용 서버로 만드는 작업을 했다. REST API를 걷어내는 것이 첫 번째 일이었다. index.ts 410줄, 레이트 리미터, Hono 의존성 두 개. 지우고 나니 패키지가 한결 가벼워졌다. 사용자가 직접 아트를 등록하고 삭제하는 기능도 빼버렸다. 큐레이션되지 않은 콘텐츠는 이 프로젝트의 방향과 맞지 않았다.

빼낸 자리에 새로운 것들을 넣었다. 유니코드 텍스트 스타일링 도구가 하나, 박스 프레임이 하나, 프로그레스 바가 하나. 모두 외부 의존성 없이 유니코드 문자 매핑만으로 동작한다. 이미지 컨버터에는 브라유 점자 모드를 추가했다. 한 글자당 2x4 도트를 쓸 수 있으니 해상도가 여덟 배로 올라간다.

다이어그램 타입도 일곱 개에서 열한 개로 늘렸다. 클래스 다이어그램, ER 다이어그램, 마인드맵, 간트 차트. 개발자들이 터미널에서 실제로 그리고 싶어하는 것들이다. 테스트는 97개 전부 통과했고, 빌드된 dist에는 테스트 파일이 더 이상 섞이지 않는다.

결과적으로 1,221줄을 지우고 1,068줄을 새로 썼다. 도구 수는 열두 개가 되었고, 버전을 0.5.0으로 올려서 npm에 퍼블리시했다. README도 REST API 관련 내용을 모두 걷어내고 새 기능 중심으로 다시 썼다. 줄이 좀 더 빠졌으면 좋겠지만, 오늘은 여기까지다.

클라우드 비용 분석을 하다가 이상한 숫자를 발견했다. 월 23테라바이트의 데이터 전송. MAU 30만짜리 서비스치고는 지나치게 높았다. CDN을 잘 구성해뒀는데도 그랬다. 청구서를 뜯어보니 CDN이 처리하는 18테라바이트와는 별개로, 스토리지에서 14테라바이트가 직접 빠져나가고 있었다. 기가바이트당 11센트. 한 달에 천오백 달러가 조용히 새고 있었던 셈이다.

어디서 새는지 찾아야 했다. 세 개의 코드베이스를 뒤졌다. 원인은 뜻밖에 단순했다. 이미지 URL을 만들 때 스토리지의 원본 주소를 그대로 내보내고 있었던 것이다. CDN 주소로 바꿔주는 코드가 빠져 있었다. 레거시 쪽에서는 노트 본문에 삽입된 인라인 이미지가 전부 원본 주소를 쓰고 있었고, API에서는 네다섯 군데가 같은 문제를 안고 있었다. 프론트엔드에는 하드코딩된 스토리지 URL이 하나 있었다.

수정 자체는 어렵지 않았다. 이미지 경로를 반환하는 곳마다 CDN 변환 함수를 끼워 넣으면 됐다. 레거시 PHP에서 한 군데, API 타입스크립트에서 네 파일, 프론트엔드에서 상수 하나. 테스트를 돌리니 한 곳에서 깨졌다. 이미 CDN도 스토리지도 아닌 외부 URL이 들어올 때를 처리하지 못했다. 조건 분기를 하나 추가해서 해결했다. 2,494개 테스트가 모두 통과한 뒤 세 개의 PR을 올렸다.

월 천사백 달러 정도가 줄어들 것으로 보인다. 연으로 치면 이천삼백만 원쯤 된다. 데이터베이스의 데이터 자체는 깨끗했다. 상대 경로로 잘 저장되어 있었다. 문제는 그 경로를 URL로 바꾸는 코드 한 줄에 있었다. 비용 최적화라는 것이 대개 그렇다. 거창한 아키텍처 변경이 아니라, 놓친 한 줄을 찾는 일이다.

오디오 청취 진행률을 추적하는 테이블을 새로 만들었다. 기존에 유저별 노트 속성을 담는 테이블이 1,300만 행이 넘어가고 있어서, 오디오 데이터는 별도 테이블로 분리하기로 했다. 마이그레이션, 모델, 컨트롤러, 라우트. 기존 텍스트 읽기 진행률 패턴을 그대로 따랐다. 진행률은 단방향으로만 갱신된다. 뒤로 돌아가는 일은 없다.

PR을 올렸더니 자동 리뷰가 달렸다. 열두 개쯤 되는 제안 중 합리적인 것은 하나뿐이었다. 0에서 100 사이 범위 클램핑. 나머지는 트랜잭션 래핑, 레이트 리미팅, 이벤트 발행 같은 것들이었다. 기존 코드에도 없는 것을 새 코드에만 넣으면 일관성이 깨진다. 오버엔지니어링의 유혹은 리뷰에서 가장 자주 나타난다.

머지 후에 실제로 데이터가 적재되지 않는다는 보고가 들어왔다. 원인은 단순했다. 백엔드만 만들고 프론트엔드 호출 코드를 빠뜨린 것이다. 오디오 플레이어의 timeupdate 이벤트에서 UI만 갱신하고 서버에는 아무것도 보내지 않고 있었다. API 라우트를 추가하고, 기존 텍스트 진행률과 같은 방식으로 토큰 인증 래퍼를 만들었다. 재생 중 30초 간격으로 전송하고, 페이지를 떠날 때는 sendBeacon으로 마지막 값을 보낸다. 배관 공사에서 파이프를 깔아놓고 수도꼭지를 안 단 셈이었다.

I worked on turning artscii into a pure MCP server. The first task was removing the REST API. That meant index.ts at 410 lines, the rate limiter, and two Hono dependencies. Once they were gone, the package felt noticeably lighter. I also removed the user-submitted art features — submit and delete. Uncurated content didn't fit the direction of this project.

In the space left behind, I added new things. A Unicode text styling tool, a box frame tool, and a progress bar tool. All of them work purely through Unicode character mapping, with no external dependencies. I added a braille mode to the image converter as well. Each character covers a 2x4 dot grid, so resolution goes up eightfold.

Diagram types grew from seven to eleven. Class diagrams, ER diagrams, mind maps, Gantt charts. The kinds of things developers actually want to draw in a terminal. All 97 tests pass, and the compiled dist directory no longer includes test files.

In total, 1,221 lines removed, 1,068 lines added. The tool count is now twelve, and I bumped the version to 0.5.0 and published to npm. The README was rewritten to drop all REST API references and focus on the new features. I wish a few more lines could have been trimmed, but that's enough for today.

While analyzing cloud costs, I found an odd number. Twenty-three terabytes of data transfer per month. Far too high for a service with 300,000 monthly active users. The CDN was properly configured, and yet. Breaking down the bill, I saw that apart from the 18 terabytes the CDN was handling, another 14 terabytes were leaking straight from the storage layer. Eleven cents per gigabyte. Fifteen hundred dollars a month, quietly draining away.

I had to find the leak. I searched through three codebases. The cause turned out to be surprisingly simple. When generating image URLs, the code was returning the raw storage address instead of the CDN address. The conversion step was just missing. On the legacy side, every inline image embedded in note content was using the raw storage URL. The API had the same problem in four or five places. The frontend had one hardcoded storage URL.

The fix itself wasn't difficult. I just needed to insert a CDN conversion function wherever image paths were returned. One spot in the legacy PHP, four files in the API TypeScript, one constant in the frontend. Running tests, one broke — it couldn't handle external URLs that were neither CDN nor storage. I added a single conditional branch to fix it. All 2,494 tests passed, and I opened three PRs.

The savings should be around fourteen hundred dollars a month. Roughly twenty-three million won per year. The data in the database itself was clean — stored as relative paths, exactly as it should be. The problem was in a single line of code that converted those paths into URLs. That's how cost optimization usually goes. Not a grand architectural overhaul, but finding the one line that was missed.

I created a new table for tracking audio listening progress. The existing table for per-user note properties had grown past 13 million rows, so I decided to keep audio data in a separate table. Migration, model, controller, route. I followed the existing text reading progress pattern exactly. Progress only moves forward. It never goes back.

After opening the PR, an automated review came in. Out of about a dozen suggestions, only one was reasonable — clamping values to the 0-to-100 range. The rest were things like transaction wrapping, rate limiting, event dispatching. Adding things to new code that don't exist in the existing code breaks consistency. The temptation to over-engineer shows up most often in reviews.

After the merge, a report came in that data wasn't actually being stored. The cause was simple. I had built the backend but forgotten the frontend call. The audio player's timeupdate event was only updating the UI, sending nothing to the server. I added an API route and created a token authentication wrapper in the same pattern as the text progress endpoint. It sends every 30 seconds during playback, and fires a sendBeacon with the last value when leaving the page. It was like laying down all the pipes but forgetting to attach the faucet.
