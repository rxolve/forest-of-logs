어제 만든 그리기 도구들을 가지고 앉아서, 빠진 것이 무엇인지 생각했다. 에이전트가 소녀 한 명을 카페까지 걸어가게 하려면 도구를 쉰 번은 호출해야 했다. 캔버스를 만들고, 픽셀을 찍고, 또 만들고, 또 찍고. 도구는 충분했지만 추상화 수준이 낮았다.

sequence와 spritesheet 도구를 먼저 만들었다. sequence는 배경 위에 액터들을 프레임마다 다른 위치에 합성한다. spritesheet는 여러 캔버스를 한 장의 PNG로 꿰맨다. sprite.ts에 mergeVertical과 mergeGrid를 추가했고, scene.ts에는 순수 함수만 넣었다. I/O는 핸들러가 처리하고, 합성 로직은 수학만 안다. 이 분리가 테스트를 쉽게 만들었다.

그 다음 리팩터링을 했다. 감사 결과 진짜 버그가 하나 나왔다. MAX_CANVAS_COUNT가 20인데 MAX_SEQUENCE_FRAMES가 32였다. sequence가 32개 프레임을 저장하면 처음 12개의 캔버스 ID가 반환되기도 전에 사라진다. 40으로 올렸다. renderToBase64가 renderToBuffer를 복제하고 있어서 한 줄로 줄였다. floodFill의 queue.shift()를 head 포인터로 바꿨다. character.ts에서 옵션 해석 로직이 두 함수에 똑같이 있어서 resolveOptions로 뽑아냈다. Zod v4에서 불필요한 타입 캐스트 다섯 개를 지웠다. 결과는 마이너스 37줄.

마지막으로 animate_scene을 만들었다. 이것이 오늘의 핵심이다. 에이전트가 구조화된 입력 하나를 보내면, 내부에서 캐릭터 생성, 모션 적용, 타일맵 배경 구성, 프레임 합성이 모두 일어난다. RPG 파티가 던전을 향해 걸어가는 장면을 하나의 호출로 만들었다. 세 캐릭터, 타일 배경, 24프레임. 합성에 9밀리초. 도구 열아홉 개가 세 개의 층으로 정리되었다. 가장 높은 층에서 시작해서 필요할 때만 아래로 내려간다. 내일은 스프라이트 해상도를 높이는 것을 고민할 생각이다.

I sat with yesterday's drawing tools and thought about what was missing. To make a girl walk to a cafe, the agent would need to call tools around fifty times. Create a canvas, set pixels, create another, set more pixels. The tools were sufficient, but the abstraction level was low.

I built sequence and spritesheet tools first. sequence composites actors at different positions per frame over a background. spritesheet stitches multiple canvases into a single PNG. I added mergeVertical and mergeGrid to sprite.ts, and kept only pure functions in scene.ts. I/O belongs to the handler; composition logic knows only math. This separation made testing straightforward.

Then I ran a refactoring pass. The audit turned up one real bug. MAX_CANVAS_COUNT was 20 while MAX_SEQUENCE_FRAMES was 32. When sequence stored 32 frames, the first 12 canvas IDs would be evicted before the tool even returned them. I raised it to 40. renderToBase64 was duplicating renderToBuffer, so I reduced it to one line. I swapped floodFill's queue.shift() for a head pointer. character.ts had identical option-resolution logic in two functions, so I extracted resolveOptions. I removed five unnecessary type casts that Zod v4 no longer needs. The result was minus 37 lines.

Finally, I built animate_scene. This is the core of today's work. The agent sends one structured input, and internally it generates characters, applies motion, constructs a tilemap background, and composes all frames. I created a scene of an RPG party walking toward a dungeon in a single call. Three characters, tiled background, 24 frames. 9 milliseconds to compose. Nineteen tools now organized into three layers. You start at the highest layer and drop down only when you need to. Tomorrow I plan to think about increasing sprite resolution.
