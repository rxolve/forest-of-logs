artscii에는 이미지를 ASCII 아트로 바꾸는 Python 스크립트가 있었다. Pillow로 그레이스케일 변환을 하고, 픽셀마다 밝기를 읽어 문자 램프에 대응시키는 방식이었다. 잘 동작했지만 배치 전용이었고, 런타임에 에이전트가 이미지 URL을 던지면 즉시 변환해주는 기능은 없었다. 그래서 같은 알고리즘을 Node.js의 sharp로 포팅하기로 했다.

포팅 자체는 어렵지 않았다. flatten으로 알파 채널을 흰 배경에 합성하고, grayscale, normalise, resize를 거쳐 raw 버퍼를 꺼낸 뒤 픽셀 루프를 도는 구조다. Python의 `int()` 가 truncate인 반면 처음에 `Math.round`를 썼다가, 리뷰에서 1행 차이가 날 수 있다는 지적이 나와 `Math.floor`로 맞췄다. 사소하지만 "1:1 포팅"이라고 말하려면 이런 경계값이 정확해야 한다.

REST 엔드포인트와 MCP 도구 양쪽으로 노출했다. REST 쪽은 zod 없이 수동 검증을 해야 하는데, 첫 버전에서 `url`이나 `gamma` 같은 필드의 타입 체크를 빠뜨렸다. MCP 쪽은 zod 스키마가 알아서 걸러주니까 괜찮았지만, REST에서 `{"url": 123}`을 보내면 sharp까지 내려가서야 터지는 상황이었다. 리뷰를 두 번 거치면서 SSRF 방어, fetch 크기 제한, private IP 차단, save 필드 검증까지 하나씩 채워넣었다. DNS rebinding이나 리다이렉트 체인 우회 같은 건 현재 규모에서는 수용하기로 했다.

결국 0.2.0으로 버전을 올리고 [npm에 배포](https://www.npmjs.com/package/artscii)했다. `npx -y artscii`로 MCP 서버가 뜨고, convert 도구로 이미지 URL을 넘기면 64w와 32w 두 가지 크기의 ASCII 아트가 돌아온다. Python 스크립트는 그대로 남겨두었다. 빌트인 아트를 일괄 재생성할 때는 여전히 쓸모가 있으니까.

artscii already had a Python script that converted images to ASCII art. It used Pillow to grayscale the image and mapped each pixel's brightness to a character ramp. It worked well, but only as a batch tool. There was no way for an agent to hand over an image URL at runtime and get ASCII art back instantly. So I decided to port the same algorithm to Node.js using sharp.

The port itself wasn't difficult. Flatten the alpha channel onto a white background, run grayscale, normalise, and resize, then pull out the raw buffer and loop through pixels. Python's `int()` truncates toward zero, but I initially used `Math.round`. The review caught that this could produce a one-row difference, so I switched to `Math.floor`. A small thing, but if you're going to call it a "1:1 port," these boundary values need to be exact.

I exposed the converter through both a REST endpoint and an MCP tool. The REST side required manual validation without zod, and in the first version I missed type checks on fields like `url` and `gamma`. The MCP side was fine since zod schemas handled it, but on REST, sending `{"url": 123}` would pass through until sharp blew up. Over two rounds of review, I added SSRF protection, fetch size limits, private IP blocking, and save field validation one by one. Things like DNS rebinding and redirect chain bypasses were accepted as tolerable at the current scale.

In the end I bumped the version to 0.2.0 and [published to npm](https://www.npmjs.com/package/artscii). Run `npx -y artscii` to start the MCP server, pass an image URL to the convert tool, and you get ASCII art back in both 64w and 32w sizes. The Python script stays. It's still useful for batch-regenerating the built-in arts.
